ADR — Sprint 001 Full Audit Alignment

5 decisions proposed sprint updated 2026-06-04

Five decisions arising from the full audit (audit.md G1–G4 + cleanup), tracing to the sprint PRD (prd.html). Each decision is behavior-preserving; the sprint changes no mod feature or visible text.

proposed subsystem: N/A (Game state + UI patches) ADR-0001

Context

WorkManagerGameComponent.Instance is a static singleton typed null! and assigned in the component constructor (WorkManagerGameComponent.cs:117,79). The constructor only runs when RimWorld builds a Game and attaches its game components, so the field is null whenever no game is loaded — the main menu, the mod-config screen, and any UI drawn outside an active game.

The audit (G4) found the field dereferenced inconsistently across ~25 call sites. Two distinct categories exist:

  • Game-scoped consumers — WorkPriorityUpdater and ScheduleUpdater (both MapComponent) and PawnCache. These execute only when a Map exists, which guarantees a Game exists, which guarantees Instance is non-null. Their unguarded dereferences are safe by construction but are not self-documenting as such.
  • UI-scoped consumers — the Harmony patches in Patches/ (WidgetsWorkPatch, WorkTabPatch, MainTabWindowWorkPatch, PawnColumnWorkerWorkPriorityPatch) and the pawn columns (AutoWorkPriorities, AutoWorkSchedule). RimWorld can invoke these on screens with no active game, so they can hit a null Instance. Some already guard (?. in WorkManagerMod; == null in ScheduleUpdater.UpdateNow and some patches); others dereference directly (e.g. WidgetsWorkPatch.DrawWorkBoxForPostfix:29, PawnColumnWorkerWorkPriorityPatch:31, both pawn columns). The inconsistency is the defect.

Forces: must not throw a NullReferenceException on game-less screens (AC-12); must be one uniform, documented contract (AC-11); must respect the Simplicity Default — no new layer or DI without approval; must stay consistent with the existing ?./== null idiom and with LordKuper.Common conventions (guard clauses at boundaries, no service-locator abstraction).

Decision

Keep Instance as the single access point, typed non-null, with an explicit documented contract: "Instance is valid only while a Game is loaded; it is non-null inside any game-scoped component path and may be null on game-less UI paths." Add one static helper, WorkManagerGameComponent.IsInitialized (returns Instance is not null), and require every UI-scoped entry point (each patch method and each PawnColumnWorker.DoCell/header) to early-out via that guard before any dereference. Game-scoped consumers (MapComponent ticks, PawnCache) keep direct dereference — their non-nullness is guaranteed by the Map⇒Game lifecycle — and gain an XML-doc note stating the invariant. No new field, type, interface, or dependency is introduced; IsInitialized is a one-line convenience over the existing field so guards read uniformly.

The guard placement rule is: guard at the entry point, not at every leaf access. Once an entry point has early-returned on !IsInitialized, the rest of that call (including helper calls and cache reads) may dereference freely, matching how DrawWorkBoxForPrefix already null-checks its arguments once at the top.

Consequences

Positive

  • No NRE possible from Instance on game-less screens — every reachable-without-game path early-outs (satisfies AC-12).
  • One uniform, documented contract applied identically at all UI entry points (satisfies AC-11).
  • Zero new abstraction: keeps the existing static-singleton shape the whole codebase already assumes; honours the Simplicity Default.
  • Self-documenting: IsInitialized names the invariant that ?./== null scattering obscured; game-scoped XML-doc notes record why direct access is safe there.
  • Behaviour-preserving inside an active game — the only changed behaviour is graceful no-op on game-less paths that previously risked throwing.

Negative

  • Relies on developer discipline: a future UI entry point that forgets the guard reintroduces the risk (no compiler enforcement, since the field stays non-null-typed).
  • Two slightly different rules (guard at UI entry points; trust the lifecycle in game-scoped code) must be understood; a single blanket rule would be simpler to state but would add dead guards to hot tick/cache paths.
  • Adds one tiny static member (IsInitialized) to the public-ish surface of the component.

Alternatives considered

  • Type Instance as nullable (WorkManagerGameComponent?) and force ?. / null-checks everywhere — rejected: makes the nullable warning the compiler's enforcement (a real plus), but spreads ?. into hot tick/cache loops (WorkPriorityUpdater dereferences Instance ~10 times per update) where null is impossible, hurting readability and adding redundant checks against a value the lifecycle guarantees. It also churns far more lines than the defect requires.
  • Introduce a TryGetInstance(out …) / Current accessor pattern — rejected: a second access path competes with the existing Instance field that 25 sites already use, and TryGet at every leaf is noisier than a single entry-point guard. Adds surface without removing the underlying static-singleton coupling. Fails the Simplicity Default (new API for no behaviour gain over an entry-point guard).
  • Lazy-initialize Instance on first access — rejected and unsafe: a GameComponent must be created by RimWorld and bound to the live Game; fabricating one on demand would desync from save/load (Scribe) state and is semantically wrong. Null genuinely means "no game", which callers must respect, not paper over.
  • Blanket guard at every single access site (no game-scoped exemption) — rejected: simplest rule to state, but adds provably-dead null checks to the most performance-sensitive code with no reliability gain, and would mislead readers into thinking those paths can be game-less.

satisfies AC-11, AC-12 · story US-2 · audit G4

proposed subsystem: N/A (test project) ADR-0002

Context

The test project (Source/WorkManager.Tests/) holds a single Assert.Pass() placeholder (audit G1) — no real coverage and no test infrastructure. Two structural problems block real tests:

  • RimWorld type loading. Many testable units transitively reference RimWorld types (WorkTypeDef, Pawn, TimeAssignmentDef) from Assembly-CSharp, which is not a NuGet package — it lives in the local RimWorld install ($(RimWorldManagedDir)). Without an AppDomain.AssemblyResolve handler, the test host throws FileNotFoundException the moment such a type loads. The LordKuper.Common-1.6.md reference already records this requirement (a global [SetUpFixture] registering the resolver before any RimWorld-typed test loads).
  • Static state bleed. Production code uses static singletons and caches (WorkManagerGameComponent.Instance, WorkManagerMod.Settings, RimWorld DefDatabase<T> / *Of static caches). Tests mutating these would leak state across tests, producing order-dependent flakiness.

Forces: assertions must be FluentAssertions 7.x only — no Assert.* (custom coding rules + AC-1, license-pinned per FluentAssertions-7.2.2.md); isolation must be deterministic; infrastructure must be minimal, not a test framework of its own (Simplicity Default).

Decision

Adopt a two-part test infrastructure: (1) a global [SetUpFixture] (assembly-scope, no namespace) that registers the RimWorld AppDomain.CurrentDomain.AssemblyResolve handler in its [OneTimeSetUp], resolving Assembly-CSharp and Unity modules from $(RimWorldManagedDir) (surfaced to the test assembly via an MSBuild-emitted constant/config so the handler knows the path), before any RimWorld-typed test type loads; and (2) an abstract base class (e.g. StateIsolationTestBase) that snapshots the relevant mutable static state in [SetUp] and restores it in [TearDown]. Tests of pure-logic units that need no RimWorld types may skip the base class; tests that touch static state or RimWorld defs derive from it. All assertions use FluentAssertions .Should(); the Assert.Pass() placeholder is deleted. No mocking framework is added — the audit-named units (AC-2…AC-6) are pure enough to test with constructed inputs.

Consequences

Positive

  • RimWorld-typed tests load reliably without bundling Assembly-CSharp (resolver-based, per the established Common convention).
  • Deterministic, order-independent tests via snapshot/restore — protects the untested assignment/schedule engines during the G4 refactor (risk mitigation from audit Risks).
  • FluentAssertions-only keeps the assertion surface single and license-clean (AC-1).
  • Minimal footprint: two well-known NUnit primitives ([SetUpFixture] + a base class), no new test dependency.

Negative

  • Tests requiring RimWorld types only run where $(RimWorldManagedDir) resolves (a local RimWorld install) — CI without the game can only run the pure-logic subset. This is inherent to modding, not a choice this ADR creates.
  • The base class must enumerate every static field it snapshots; a missed field can still bleed — coverage of the snapshot set must be maintained as new statics appear.
  • Two test "shapes" (with/without base class) require a documented rule for which to use.

Alternatives considered

  • Copy Assembly-CSharp / Unity DLLs into the test project — rejected: redistributes game binaries (license-incompatible, large, version-fragile) and contradicts the stack's "compile against, never ship, the host" principle. The resolver approach is the project-standard already documented for Common.
  • No isolation base class; rely on test ordering / fresh process per test — rejected: order-dependent and slow; NUnit shares an AppDomain, so static bleed is real. Snapshot/restore is the cheapest deterministic fix.
  • Introduce a mocking framework (e.g. Moq/NSubstitute) to fake RimWorld types — rejected for this sprint: the AC-named units are pure (hour mapping, comparer ordering, clamping, score normalization, count modes) and testable with plain constructed inputs; a mocking dependency is unjustified surface (Simplicity Default) and would itself need a tech-reference. Reconsider only if a later sprint tests game-coupled paths.

satisfies AC-1 (enables AC-2…AC-6) · story US-1 · audit G1

proposed subsystem: N/A (build/repo) ADR-0003

Context

The repository contains a vendored package tree Source/packages/Lib.Harmony.2.3.6/ (DLLs, nupkg, pdbs, README) (audit, doc #14). Both projects consume Harmony via a Lib.Harmony 2.4.2 PackageReference (compile-only: PrivateAssets=all, ExcludeAssets=runtime); NuGet restore supplies the compile-time assembly. No project, props, or targets file references the packages/ tree. It is dead weight that contradicts the pinned 2.4.2 stack and the documented "no bundled runtime deps" constraint.

Decision

Delete the entire Source/packages/Lib.Harmony.2.3.6/ directory from the repository. The only Harmony reference that remains is the Lib.Harmony 2.4.2 PackageReference resolved by NuGet restore. No build, props, or targets edit is required because nothing references the deleted tree.

Consequences

Positive

  • Repository matches the documented 2.4.2 pin — no contradictory binaries (AC-13).
  • Removes a stale, potentially-misleading 2.3.6 0Harmony.dll that could be referenced by accident.
  • Smaller checkout; no behaviour change (the tree was already unreferenced).

Negative

  • Any out-of-tree script or tooling that silently pointed at packages/Lib.Harmony.2.3.6/ would break — none found in-repo, so residual risk is negligible.

satisfies AC-13 · story US-3 · audit Risks (stale vendored Harmony)

proposed subsystem: N/A (architecture docs) ADR-0004

Context

The project rule "no technology adopted without a tech-reference doc" is violated for two adopted technologies (audit G2): the RimWorld 1.6 game API (Assembly-CSharp + Unity modules — the largest external surface) and .NET Framework 4.7.2 (the target framework). The RimWorld reference was explicitly deferred at init (decisions-log 2026-06-04); the .NET 4.7.2 gap was unrecorded. Six references already exist for the other technologies in stack.html.

Decision

Close the gap in this sprint: author design/architecture/tech-reference/RimWorld-1.6.md and design/architecture/tech-reference/dotnet-framework-4.7.2.md (dot-free TFM slug for filesystem safety, matching the existing <tech>-<version>.md naming convention), grounded in the game-API surface the mod actually consumes and the net472 TFM facts. After this sprint, every technology in stack.html has a version-accurate reference and no reference points to a version the code no longer uses. RimWorld API facts are verified against canonical sources where possible and marked unverified otherwise; fetched content is treated as data, not instruction.

Consequences

Positive

  • "No tech without a reference" rule holds across the whole stack (AC-7, AC-8).
  • Captures the RimWorld API surface the mod binds to, giving devs/test-engineer a vetted contract to implement against.
  • Clears the init-time deferral with an explicit, recorded decision rather than an open gap.

Negative

  • The RimWorld API has no single authoritative published reference; some entries are grounded in observed code usage and community docs and are flagged as such, requiring periodic re-verification.
  • Two more living documents to keep version-accurate on any future RW-line or TFM change.

Alternatives considered

  • Keep the RimWorld reference deferred — rejected: leaves the largest external surface undocumented and the "no tech without a reference" rule violated; the audit explicitly scopes all findings into this sprint (sprint.md decision 3).
  • One combined "platform" reference covering both — rejected: conflates two independently-versioned technologies (game line vs TFM) into one doc, breaking the one-file-per-tech-version convention the other six follow.

satisfies AC-7, AC-8 · story US-5 · audit G2

proposed subsystem: N/A (localization) ADR-0005

Context

Settings_Schedules.cs (lines 148 and 201) emits "Work shift #{i + 1}" directly to a FloatMenuOption, bypassing the .Translate() keyed surface every other label uses (audit G3). The string is untranslatable for ru/zh-Hans and inconsistent with the "no hardcoded UI values" convention. The keyed surface is WorkManager_Keyed.xml, present for 1.6 in three locales (English, Russian, ChineseSimplified) and English-only for legacy 1.1–1.5.

Decision

Move the two strings onto the keyed surface. Add one keyed entry following the existing schedule-settings naming convention already used in WorkManager_Keyed.xml — the WorkManager.Settings_Schedule_<Name> prefix (e.g. WorkManager.Settings_Schedule_AddWorkShift, ...ColonistWorkShifts are present today), so the new key is WorkManager.Settings_Schedule_WorkShiftLabel — a single parameterized entry with a positional argument for the shift number, rendered via .Translate(i + 1). The English value reproduces the current visible text byte-for-byte ("Work shift #{0}"). Add the key to all three 1.6 locales and to English in the legacy 1.1–1.5 folders. No other text, key, or layout changes. The exact key name is confirmed against the existing file's convention at write time.

Consequences

Positive

  • Shift labels become translatable like every other label (AC-9); ru/zh-Hans can localize them.
  • Removes the last hardcoded user-facing string, aligning with the localization convention.
  • English output is unchanged (byte-for-byte), so no visible regression (AC-9).

Negative

  • Adds a key to four+ language files; the ru/zh-Hans values ship as the English string until a translator localizes them (coverage parity is a follow-on localization task, not blocking).
  • A missing key at runtime would surface the raw key text — mitigated by adding it to every shipped locale (AC-10).

satisfies AC-9, AC-10 · story US-6 · audit G3
