Work Manager — Sprint 001 PRD (Full Audit Alignment)

6 goals 7 stories 18 AC 6 non-goals updated 2026-06-04

Problem

Work Manager is a mature, shipped RimWorld mod, but a full audit (audit.md) found the codebase and its documentation drifting from the four reference standards the project commits to: the .asd rules, the documented technology stack, the LordKuper.Common upstream contract, and the code conventions. The complex assignment and schedule engines have no automated test net, two adopted technologies lack the mandatory tech-reference docs, two user-facing strings bypass localization, the singleton WorkManagerGameComponent.Instance is dereferenced inconsistently (risking a NullReferenceException on game-less UI screens), a stale vendored Harmony 2.3.6 package contradicts the pinned 2.4.2, and the README plus mod metadata under-describe the current product. This sprint closes those gaps so the project is verifiably aligned with its own standards — without changing what the mod does or how it looks.

Goals

  1. G1

    Establish a real, standards-compliant unit-test foundation (FluentAssertions 7.x, static-state isolation, RimWorld AssemblyResolve registration) and cover the audit-named pure-logic units to the 80% floor.

  2. G2

    Achieve complete tech-reference coverage: every adopted technology has an accurate design/architecture/tech-reference/<tech>-<version>.md.

  3. G3

    Move the two hardcoded UI strings onto the keyed-localization surface across all shipped locales, with no change to the visible English text.

  4. G4

    Apply one uniform null-handling contract for WorkManagerGameComponent.Instance across every call site so no NRE can occur on screens without an active game.

  5. G5

    Remove dead/contradictory artifacts (the stale vendored Lib.Harmony 2.3.6 package tree) so the repository matches the pinned stack.

  6. G6

    Bring the documentation set (README, concept, stack, tech-references, About.xml) back into alignment with the audited code and the four standards.

User stories

Maintainer / Developer 4
  • as a maintainer, I want the pure-logic units (work-shift mapping, rule comparison/combination, dedicated-worker clamping, passion scoring, target-worker counting) covered by real unit tests with proper static-state isolation, so that I can refactor the assignment and schedule logic without silently breaking behavior. (G1; AC-1…AC-6)
  • as a maintainer, I want a single uniform contract for accessing WorkManagerGameComponent.Instance applied at every call site, so that the mod cannot throw a NullReferenceException from a UI patch or pawn column on screens where no game is active. (G4; AC-11…AC-12)
  • as a maintainer, I want the stale vendored Lib.Harmony 2.3.6 package tree removed, so that the repository contains no binaries that contradict the pinned Harmony 2.4.2 reference. (G5; AC-13)
  • as a maintainer, I want the production and test builds to compile clean (zero warnings, TreatWarningsAsErrors on) and the full test suite to pass green, so that every audit fix is verified by the build/quality gate before merge. (quality gate; AC-17…AC-18)
Architect / Standards owner 1
  • as the architecture owner, I want every adopted technology to have an accurate versioned tech-reference doc — including the currently missing RimWorld 1.6 game API and .NET Framework 4.7.2 references, so that the "no tech adopted without a reference" rule holds across the whole stack. (G2; AC-7…AC-8)
Mod player (end user) 1
  • as a non-English-speaking player, I want the work-shift labels in the schedule settings to come from the localization surface like every other label, so that they can be translated for my locale instead of always appearing in English. (G3; AC-9…AC-10)
Prospective user / Reader of docs 1
  • as a prospective user or contributor reading the project documentation, I want the README and mod metadata to accurately list the supported RimWorld versions, the scheduling capability, and the Harmony + LordKuper.Common dependencies, so that I understand what the mod does and what it needs before installing or contributing. (G6; AC-14…AC-16)

Non-goals

  • No UI/UX changes: this is an audit/refactor sprint. No UX-spec or design-system is produced or assumed; the work-shift localization (AC-9) preserves the exact visible English text.
  • No new mod features or behavior changes to the assignment or schedule engines beyond the uniform null contract (AC-11) — the refactor must be behavior-preserving.
  • No editing of the LordKuper.Common upstream library; it is consumed as an integration contract only.
  • No modification of ASD workflow infrastructure (.asd/rules/, .asd/templates/, .claude/, CLAUDE.md).
  • No full project requirements document (design/product/requirements.html): deferred to a future requirements sprint per the audit migration plan.
  • No tests for the live-game-dependent engine paths (multi-pass priority assignment, schedule scoring) in this sprint — only the audit-named pure-logic units are covered; broader engine coverage is out of scope here.
